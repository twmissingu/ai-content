# 通用排版与格式规范（基础层）

> 所有平台共享的排版设计原则。平台特化文件（formatting_wechat.md、formatting_xhs.md）继承本文件并声明例外。
> 基于 UI 设计最佳实践，面向移动端阅读场景。

## 核心原则

### 1. 可读性优先

- 字号不小于 13px（老年用户需 14px+）
- 正文颜色不用纯黑 `#000000`，用 `#3f3f3f` 或 `#333333` 降低刺眼感
- 行高至少 1.5 倍，给文字"呼吸空间"

### 2. 视觉层级清晰

- 通过字号、加粗、颜色建立 3 级层级：大标题 > 小标题 > 正文
- 同级元素样式必须统一（不能一段加粗一段不加粗）
- 层级差异要明显，不能"差不多大"

### 3. 留白即设计

- 段落之间必须有间距，不能连成文字墙
- 每段不超过 3-5 行，超过则拆分
- 移动端首行不缩进（顶格更整洁，技术内容尤其如此）

### 4. 节奏感

文章需要"呼吸"，交替使用紧凑段落和松散元素：

```
紧凑段落（3-4行，信息密集）
  ↓
加粗金句（1行，视觉锚点）
  ↓
松散列表（3-5项，留白充足）
  ↓
紧凑段落（3-4行，信息密集）
```

## 加粗规范（通用）

### 什么时候加粗

- **核心判断句**：读者扫一眼就能抓住的重点
- **对比转折**："以前…，**现在…**"
- **功能/要点定位**：每个模块的一句话定位
- **结论性语句**

### 加粗禁忌

- 不要整段加粗（等于没加粗）
- 不要连续两句话都加粗
- 加粗内容控制在 **15 字以内** 最佳
- 加粗是"视觉锚点"，不是"高亮全文"

### 好坏对比

```
好：这把"审查"从评论层推进到了执行层。
坏：以前最头疼的就是权限判断，尤其是涉及 Shell、PowerShell、git worktree 的场景。
    ↑ 太长，扫读时抓不住重点
```

## 配色原则（通用）

- 全文不超过 **3 种强调色**
- 正文颜色统一，不要花花绿绿
- 深色模式需考虑（避免深色背景 + 深色文字）
- 品牌色用于标题和重点句，全文统一

## 列表格式（通用）

- 无序列表用于正文要点
- 数字编号用于步骤/建议（1. 2. 3.）
- 列表项不超过 5 条，超过则拆分子分组
- 列表项之间不空行

## 代码与技术术语（通用）

- 命令/参数/快捷键：用行内代码 `` ` `` 包裹
  - 例：`/code-review --fix`、`Ctrl + T`、`disallowed-tools`
- 代码块：仅在需要展示多行配置或输出时使用
- 技术术语首次出现时可加代码格式，后续直接写文字

## 中英文混排规范

- 中文与英文/数字之间加半角空格
  - 好：正文字号设为 15px，推荐使用 Noto Sans SC 字体
  - 坏：正文字号设为15px，推荐使用Noto Sans SC字体
- 英文单位与数字之间不加空格：900px、2000字
- 专有名词保持官方写法：iPhone、GitHub、ChatGPT

## 移动端适配

- 图片宽度不超过屏幕（100% 宽度）
- 表格在窄屏下可能横向滚动，优先用列表替代
- 避免过长的连续英文单词（可能溢出）
- 单屏显示 35-45 字为宜（行宽适中）

## 禁止清单（通用）

- 整段加粗
- 连续 2 句以上都加粗
- 纯文字墙超过 6 行无断点
- 过度使用感叹号（全文 3-5 个上限）
- 中英文混排不加空格
- Emoji 堆砌（喧宾夺主，干扰阅读）
