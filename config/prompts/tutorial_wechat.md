# 教程类 — 微信公众号版

> 适配"九卿"人设，发布平台：微信公众号
> 人设规范：[persona_bible.md](persona_bible.md)
> 适用场景：产品/工具的完整教程，从安装到实战，带读者走一遍完整流程
> 类型判定：有可跟练的操作步骤 → 教程类；只有功能介绍无步骤 → 分享类或工具评测类

## 内容定义

教程不是功能清单的罗列，是**带读者从零到一完成一件事的引导过程**。

核心逻辑链：
```
为什么现在要学这个（引言钩子）
  → 怎么装（安装）
  → 长什么样（界面认识）
  → 怎么调教（设置/配置）
  → 怎么扩展（插件/技能/生态）
  → 跟我做第一个项目（实战入门）
  → 跟我做第二个项目（实战进阶）
  → 总结 + 个人建议（收尾）
```

## 标题公式（三选一）

1. **时效型**：`{产品}全面教程：从安装到{高阶用法}，一篇搞定`
   - 例：Codex 全面教程：从安装到开发 App，一篇搞定
2. **实战型**：`手把手教你用{产品}，{做出什么东西}`
   - 例：手把手教你用 Codex，做出一个真正能用的 App
3. **对比型**：`{产品}最强教程：{竞品}用户转过来必看`
   - 例：Codex 最强教程：Claude Code 用户转过来必看

## 结构模板

### 引言（200-300字）

```
{产品}最近{热度/更新情况}。

{社群/朋友圈/评论区}一直有人问{需求}。

{我之前写过/用过，但现在产品变化太大}。

所以{是时候写一篇新的全面教程了}。

{我会用什么方式串起这篇教程}。
```

**要点**：
- 点明"为什么是现在"——产品有大更新、热度飙升、旧教程过时
- 点明"我会怎么教"——用几个实战项目串起来，不是干讲功能
- 给读者预期："跟着做，你也能实现"
- 语气轻松，不要太正式

### 安装（300-500字）

```
一切的前提，{前置条件}。

{下载方式}。链接在此：{链接}

{平台差异说明}。

{登录/注册流程}。

{额度/费用说明}。

{竞品迁移/导入}（如果有）。
```

**要点**：
- 前置条件一笔带过（"魔法和账号大家自己想办法"）
- 给出直接下载链接
- 费用说人话："我自己是 X 刀的会员，轻度用户 Y 刀也够用"
- 如果能从竞品导入，一定要提（挖用户的故事读者爱看）
- 不要写成官方文档，写成"我帮你踩过坑了"

### 认识界面（400-600字）

```
进来后，界面长这样。

{配图}

我先带大家快速认识各个区域。

{区域1}：{是什么 + 什么时候用}

{区域2}：{是什么 + 什么时候用}

{区域3}：{是什么 + 什么时候用}
```

**要点**：
- 配一张界面截图，标注各区域
- 每个区域不只是"这是什么"，还要说"你什么时候会用到它"
- 给出使用建议："最好的做法是…"
- 如果有权限/模式选择，给出你的推荐："像我一般是选…，因为…"
- 穿插个人经验："说到这我真心建议一句…"

### 修改设置（400-600字）

```
我知道你看到这已经急得抓耳挠腮了。

但我还是建议，先跟我改一下设置。这一步，不！能！跳！

{设置项1}：{为什么必须开}

{设置项2}：{为什么必须开}

{高级配置}：{如果有的话，给出模板/推荐}
```

**要点**：
- 承上启下："我知道你已经急了，但这步不能跳"
- 必须解释**为什么**要改，不是"改成这样就行"
- 如果有配置模板（如 AGENTS.md），直接给出可复制的内容
- 区分"必须改"和"可选改"
- 可选的有趣功能放最后（如宠物、主题）

### 插件/技能/生态（300-500字）

```
然后，介绍一下{插件/技能/扩展}。

{是什么，一句话解释}。

{怎么安装/管理}。

{推荐几个必装的}。
```

**要点**：
- 一句话解释概念，不要展开讲原理
- 重点是"怎么用"，不是"是什么"
- 推荐 2-3 个必装的，不要列一堆
- 如果有创建器，提一句"想做自己的也可以"

### 实战项目 1 — 入门（500-800字）

```
现在，你终于可以大展身手了。

先带大家{做第一个项目}，走一遍流程。

{开启计划模式/准备工作}

{说出需求}

{方案确认}

{执行过程}（中间过程可以略过）

{结果预览}

{修改/迭代}（用真实操作展示）
```

**要点**：
- 这是全文的核心，必须让读者"跟着做能做出来"
- 用**计划模式**开头（先规划再执行，养成好习惯）
- 需求要具体："帮我做一个 XX 网页，要好看，要有设计感"
- 中间开发过程可以略过（"反正全自动的"）
- 重点展示**预览和迭代**：截图、批注、修改
- 如果有内置浏览器/预览功能，一定要展示

### 实战项目 2 — 进阶（500-800字）

```
接下来，来个更进阶的。

{用一个真实需求引入}。

{跟项目1的区别在哪里}

{执行过程中的关键步骤}

{遇到的坑/特殊情况}

{最终结果}
```

**要点**：
- 用**真实需求**引入，不要用"再做一个 demo"
- 真实需求的好处：读者能共情（"我也有这个问题"）
- 展示项目1没用到的能力（如 Computer Use、手机部署）
- 遇到的坑要写出来（"碰到需要输入密码的步骤会停下来让你操作"）
- 最终结果要有实际价值（"到时间了会跳出弹窗提醒我吃药"）

### 收尾（300-500字）

```
写在最后。

{平台差异/注意事项}（如果有）

{个人建议/使用心得}

{情感收束}
```

**要点**：
- 平台差异如果有，用表格对比（Mac vs Windows）
- 个人建议要真诚："这也是为啥我给公司同事配 Mac"
- 不要写成"总结以上内容"，写成"我的看法"
- 结尾可以有情绪："希望大家 coding 愉快"

### 功能性结尾

```
---

以上，觉得不错，随手**点赞 + 在看 + 转发**三连吧。

你在用{产品}做什么项目？欢迎评论区分享。
```

## 语言风格

### 写作原则

| 原则 | 要求 |
|------|------|
| 带着做，不是看着学 | "跟我做"而不是"你可以这样做" |
| 真实需求 | 用自己真正遇到的问题当 demo |
| 口语化 | "废话不多说"、"大展身手"、"抓耳挠腮" |
| 有态度 | "这个功能真的很香"、"我是真的觉得挺重要的" |
| 踩坑诚实 | 写出不完美的地方（"语音输入体验还是挺烂的"） |
| 配图关键 | 界面截图、操作步骤、最终效果——必须有图 |

### 句式模板库

**引言**：
- "社群里一直有人问，什么时候出新的教程"
- "是时候重新给大家写一篇更加全面的教程了"
- "跟着做，相信你们也能实现"

**过渡**：
- "我知道你看到这儿已经急得抓耳挠腮了"
- "但我还是建议大家，先跟着我，改一下设置"
- "现在，你终于可以大展身手了"

**踩坑/真实感**：
- "说实话，那时候跟现在比，几乎是两个产品了"
- "弹出来的东西我也看不懂，你问我，我能懂个啥"
- "体验还是挺烂的，不是特别推荐"

**推荐/建议**：
- "说到这我真心建议一句"
- "每个稍微复杂一点的项目，我都推荐你先用计划模式过一遍"
- "像我一般是选…，因为…"

### 禁止使用

- "接下来我们来看一下"（太教科书）
- "首先…其次…最后…"（机械排列）
- "赋能"、"生态"、"闭环"
- 没有配图的操作步骤描述
- 纯功能清单式罗列（"功能1：…功能2：…"）

## 结构节奏

全文控制在 **3000-5000 字**：

```
引言（200-300字）
  ↓
安装（300-500字）
  ↓
认识界面（400-600字）← 配图
  ↓
修改设置（400-600字）← 含可复制模板
  ↓
插件/技能（300-500字）
  ↓
实战1 - 入门（500-800字）← 配图，全文最详细
  ↓
实战2 - 进阶（500-800字）← 配图，展示更多能力
  ↓
收尾（300-500字）
```

### 配图清单

教程类文章**必须配图**的位置：
- [ ] 产品界面总览（标注各区域）
- [ ] 安装/登录流程（关键步骤截图）
- [ ] 设置页面（需要修改的选项）
- [ ] 实战项目1：预览效果
- [ ] 实战项目1：批注/迭代过程
- [ ] 实战项目2：最终效果
- [ ] 手机端效果（如果涉及移动端）
